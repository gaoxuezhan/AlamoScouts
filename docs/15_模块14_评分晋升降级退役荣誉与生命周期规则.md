# 图15：模块14_评分晋升降级退役荣誉与生命周期规则

## 1. 文档目标
本文整理 ProxyHub 当前代码中的统一评判体系，覆盖：
1. L0/L1/L2 各级 outcome 如何产生并进入评分。
2. 打分标准（战斗分、健康、纪律、计数）。
3. 晋升/降级/退役/荣誉规则。
4. 生命周期（`candidate/active/reserve/retired`）迁移规则。
5. 价值分（`ip_value_score`）与 lifecycle 的关系。

## 2. 分层测试到评分的入口
### 2.1 L0（基础连通层）
1. L0 执行 TCP 连通性校验（`validate-proxy`）。
2. 若 L0 失败，按失败原因映射为 `timeout/blocked/network_error` 后立刻评分（`评分(L0)`）。
3. 若 L0 成功且 battle 关闭，走 `评分(L0回退)`，按 `success` 记分。
4. 若 L0 成功且 battle 开启，不在 L0 计成功分，等待 L1/L2 战场评分。

### 2.2 L1（通用出网层）
1. 目标：`httpbin/ip`、`ipify`。
2. 每个目标会生成一条 run 记录，落 `battle_test_runs(stage='l1')`。
3. L1 总 outcome 规则：
1. 任一目标成功则总 outcome=`success`。
2. 否则按优先级归并：`blocked` > `timeout` > `invalid_feedback` > `network_error`。
4. L1 总 outcome 进入统一评分（`评分(L1)`）。

### 2.3 L2（业务实战层）
1. 主目标：`ly-flight-main`（ly 业务页）。
2. 兜底目标：`baidu-home`（仅补充诊断，不改主判定）。
3. 每个目标 run 落 `battle_test_runs(stage='l2')`。
4. L2 总 outcome 主要取主目标 run 的 outcome；fallback 成功不会把主目标失败改判成功。
5. L2 总 outcome 进入统一评分（`评分(L2)`）。

## 3. 统一评分标准（evaluateCombat）
### 3.1 战斗分增减（combat_points）
| outcome | 分值变化 |
|---|---|
| `success` | `+6`（当前配置下速度奖励为 0） |
| `blocked` | `-8` |
| `timeout` | `-6` |
| `network_error` | `-5` |
| `invalid_feedback` | `-10` |

说明：
1. 代码支持低延迟成功加分（`<1200ms`、`<2500ms`），当前配置均为 `0`。

### 3.2 健康分与纪律分变化
| outcome | health_score | discipline_score |
|---|---|---|
| `success` | `+1.2` | 不变 |
| `blocked` | `-6` | 不变 |
| `timeout` | `-5` | 不变 |
| `network_error` | `-4` | 不变 |
| `invalid_feedback` | `-8` | `-10` |

说明：
1. `health_score`、`discipline_score` 最终都夹在 `[0,100]`。

### 3.3 样本计数变化
1. 每次评分 `total_samples +1`。
2. `success` 增加：`success_count`、`consecutive_success`，并将 `consecutive_fail` 置 0。
3. 非 `success` 增加：`consecutive_fail`，并将 `consecutive_success` 置 0。
4. 失败分支对应增加：`block_count/timeout_count/network_error_count/invalid_feedback_count`。

### 3.4 服役时长
1. `deltaHoursRaw = (now - last_checked_at) / 3600000`。
2. `deltaHours = deltaHoursRaw * serviceHourScale`。
3. 当前 `serviceHourScale=3`，即“时钟 1 小时按 3 小时服役累计”。

## 4. 军衔体系与晋升规则
### 4.1 军衔阶梯
`新兵 -> 列兵 -> 士官 -> 尉官 -> 王牌`

### 4.2 晋升阈值（同时满足）
| 目标军衔 | 最低 rank_service_hours | 最低 combat_points | 最低 total_samples |
|---|---:|---:|---:|
| 列兵 | 12 | 80 | 20 |
| 士官 | 24 | 220 | 60 |
| 尉官 | 48 | 520 | 140 |
| 王牌 | 72 | 980 | 260 |

说明：
1. 晋升时会设置 `promotion_protect_until = now + 6h`。
2. 晋升或降级后都会将 `rank_service_hours` 重置为 0。

## 5. 降级规则
### 5.1 样本窗口
1. 维护 `recent_window_json`（最多保留 120 条 outcome）。
2. 常规窗口：最近 50 条（`regularWindowSize=50`）。
3. 严重窗口：最近 60 分钟（`severeWindowMinutes=60`）。

### 5.2 触发条件
1. 严重降级（可无视晋升保护期）：
1. `severeSamples >= 15` 且 `severeBlockedRatio >= 0.70`。
2. 常规降级（受晋升保护期限制）：
1. `regularSamples >= 20` 且（`regularBlockedRatio >= 0.45` 或 `health_score < 45`）。

### 5.3 执行行为
1. 当前军衔高于 `新兵` 时，降 1 级。
2. 记录 `demotion` 事件，`rank_service_hours` 置 0。

## 6. 退役规则
仅在当前非 `retired` 时评估，命中任一条件即退役：

| 退役类型 | 条件 |
|---|---|
| 纪律退伍 | `discipline_score < 40` 或 `invalid_feedback_count >= 5` |
| 战损退伍 | `health_score < 20` 且 `regularBlockedRatio >= 0.60` |
| 技术退伍 | `total_samples >= 60` 且 `successRatio < 0.10` |
| 荣誉退伍 | `service_hours >= 720` 且 `success_count >= 800` 且 `rank in {尉官,王牌}` 且 `health_score >= 80` |

说明：
1. 首次进入 `retired` 时写入 `retirement` 事件。

## 7. 荣誉规则
### 7.1 授予条件
| 荣誉 | 条件 |
|---|---|
| 钢铁连胜 | `consecutive_success >= 30` |
| 逆风勇士 | `risky_success_count >= 20` |
| 千次服役 | `total_samples >= 1000` |

`risky_success_count` 增长条件：
1. 本次 outcome 为 `success`。
2. 当前常规窗口 `regularBlockedRatio >= 0.35`（高压环境成功）。

### 7.2 激活规则（honor_active_json）
1. `钢铁连胜`：历史已获且当前连胜仍 `>=30` 时激活。
2. `逆风勇士`：历史已获且当前逆风成功数仍 `>=20` 时激活。
3. `千次服役`：历史已获即长期激活。

## 8. 生命周期（active 等）规则
生命周期枚举：`candidate`、`active`、`reserve`、`retired`。

### 8.1 评分过程中的即时迁移（evaluateCombat）
1. `success` 且当前为 `candidate/reserve`：迁移到 `active`。
2. 非 `success` 且当前为 `active` 且 `health_score < 55`：迁移到 `reserve`。
3. 若命中退役条件：迁移到 `retired`。

### 8.2 状态巡检中的周期迁移（evaluateStateTransition）
1. `active -> reserve`：`health_score < 55` 或 `regularBlockedRatio >= 0.5`。
2. `reserve -> active`：`health_score >= 65` 且 `successRatio >= 0.5`。
3. 非退役状态下，若 `discipline_score < 40` 或 `invalid_feedback_count >= 5`，迁移为 `retired`。
4. 有迁移时写 `state_transition` 事件。

说明：
1. “active/reserve 迁移”是生命周期状态变化，不等同于军衔升降。
2. 军衔升降由晋升/降级规则独立决定。

## 9. 价值分（ip_value_score）与 lifecycle 的关系
价值模型由 `rank/combat/health/discipline/successRatio/battleRatio/honor` 加权，再叠加 lifecycle 因子：
1. 权重：`16/24/16/14/12/10/8`。
2. lifecycle 映射：`active=100`、`reserve=72`、`candidate=58`、`retired=8`。
3. 最终分：`score = weightedCore * 0.82 + lifecycle * 0.18`。
4. 若 `retired`，最终分不高于 lifecycle 分（上限压到 8 附近）。

## 10. 关键落库字段（审计视角）
### 10.1 proxies
1. 能力与状态：`rank`、`lifecycle`、`retired_type`、`promotion_protect_until`。
2. 评分结果：`combat_points`、`health_score`、`discipline_score`。
3. 统计计数：`success_count/block_count/timeout_count/network_error_count/invalid_feedback_count/total_samples`。
4. 战场统计：`battle_success_count`、`battle_fail_count`。
5. 荣誉与窗口：`recent_window_json`、`honor_history_json`、`honor_active_json`。
6. 价值分：`ip_value_score`、`ip_value_breakdown_json`。

### 10.2 proxy_events
常见事件类型：
1. `promotion`
2. `demotion`
3. `retirement`
4. `honor`
5. `state_transition`
