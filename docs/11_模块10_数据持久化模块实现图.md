# 图11：模块10_数据持久化模块实现图

## 1. 图示
```mermaid
flowchart TD
    A["0000_ensureDirForFile_确保目录文件逻辑"] --> B["0001_constructor_初始化实例逻辑"]
    B --> C["0002_init_初始化逻辑 (建表/索引/预编译语句)"]

    D["业务写入入口"] --> E["0010_updateProxyById_更新代理标识逻辑"]
    D --> F["0011_insertProxyEvent_写入代理事件逻辑"]
    D --> G["0012_insertRuntimeLog_写入运行时日志逻辑"]
    D --> H["0013_upsertHonor_插入更新荣誉逻辑"]
    D --> I["0015_insertRetirement_写入退伍逻辑"]
    D --> J["0016_insertPoolSnapshot_写入线程池快照逻辑"]

    K["查询入口"] --> L["0187_getProxyList_获取代理列表逻辑"]
    K --> M["0186_getEvents_获取事件逻辑"]
    K --> N["0185_getRuntimeLogs_获取运行时日志逻辑"]
    K --> O["0017_getRankBoard_获取军衔看板逻辑"]
    K --> P["0020_getLatestSnapshot_获取最新快照逻辑"]
```

## 2. 中文讲解
1. 数据层入口从 `0001_constructor_初始化实例逻辑` 开始，先通过 `0000_ensureDirForFile_确保目录文件逻辑` 确保数据库目录存在。
2. `0002_init_初始化逻辑` 负责建全部业务表与索引，包括 `proxies/proxy_events/runtime_logs/honors/retirements/pool_snapshots`。
3. 业务主链路写入统一通过 DB 类方法，不允许上层散写 SQL，便于维护事务一致性。
4. `0016_insertPoolSnapshot_写入线程池快照逻辑` 还承担历史清理职责，按保留天数删除过期快照。
5. 读接口同样统一在 DB 层，前端与 API 只消费 `get*` 方法，形成清晰分层。
6. 这种设计把“规则计算”和“数据持久化”隔离开，降低改规则时对 SQL 的耦合风险。
